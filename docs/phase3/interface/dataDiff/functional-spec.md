# Data Diff Functional Specification

## 1. Document Position

- Screen name: `dataDiff`
- Front-end implementation: `ui/admin-console/src/screens/DataDiffScreen.tsx`
- Parent orchestration: `ui/admin-console/src/App.tsx`
- Documentation stage rule:
  - This document defines current and intended screen behavior for review.
  - It must not trigger UI or backend implementation changes in the current stage.
  - Any discovered gap can only be recorded as a review item.

## 2. Screen Purpose

The `dataDiff` screen is the database-change comparison page for one selected run. Its job is to show the operator what changed before execution, after execution, and after restore.

This screen answers these operator questions:

- Which database rows were changed by this run?
- Which changes were expected vs unexpected?
- Which changes were restored successfully?
- Should I inspect raw diff payloads or trigger a re-restore action?

## 3. Operator Role

Primary users:

- QA platform operator
- test engineer validating DB side effects
- reviewer checking restore correctness after a run

Typical usage moments:

- after entering from `reportDetail`
- when validating expected data-plan outcomes
- when checking whether restore completed successfully

## 4. Screen Placement in Product Flow

The screen lives inside the common admin-console frame:

- top bar
- sidebar
- main panel: `dataDiff`

The screen is downstream of:

- `reportDetail`

The screen is a specialized leaf diagnostic page in the current flow.

## 5. Current Implementation State

Current implementation facts:

- The page is rendered by `DataDiffScreen.tsx`.
- The page depends on:
  - `snapshot`
  - `selectedRunName`
  - `title`
  - `locale`
- selected run context is resolved through `selectReportViewModel(snapshot, selectedRunName)`.
- diff rows are generated by local helper `makeDiffRows(report)`.
- `View raw JSON` is implemented: fetches raw diff data from the backend and opens an in-page drawer with three tabs (before/after/afterRestore).
- `Re-restore` is implemented: posts a restore-retry request to the backend, shows success/rejected/error status feedback, and refreshes diff data on success.

This matters for review:

- the page already has the correct information structure for diff review
- but the diff rows are still local placeholder data, not backend-owned diff results

## 7. Data Semantics by Area

### 7.1 Selected Run Context

- the page uses `selectedRunName` from app state to choose its run context

### 7.2 Local Diff Row Model

- `makeDiffRows(report)` creates the table rows locally
- row content is shaped from the selected report summary and status, not from a backend diff payload

This means:

- the current diff table is an operator-friendly placeholder model
- it is not yet a stable backend diff contract

### 7.3 Database Name Source

- DB display name is currently parsed from `snapshot.environmentConfig[0]`
- it is not tied to a run-specific database binding contract

## 10. Functional Control Responsibility Matrix

### 10.1 Hero Controls

- `View raw JSON`
  - function: inspect raw diff payloads
  - output type: in-page drawer with before/after/afterRestore tabs
  - current implementation: implemented via `GET /api/phase3/runs/{runId}/data-diff/raw`
- `Re-restore`
  - function: retry restore workflow for the selected run
  - output type: restore-control mutation with status feedback
  - current implementation: implemented via `POST /api/phase3/runs/{runId}/restore/retry`

### 10.2 Diff Table

- diff row
  - function: show field-level comparison result
  - output type: read-only diagnostic display
  - current implementation: implemented as display-only row

## 11. State Model

The screen should support:

- no selected run
- selected run loaded
- diff data loaded
- raw JSON action pending
- re-restore action pending
- restore retry success
- restore retry error

Current implementation status:

- selected run loaded or fallback title state is implemented
- raw JSON action: pending → loaded → drawer open (with error state)
- re-restore action: idle → pending → success / rejected / error (with status bar display)

## 12. Validation and Rules

Current intended comparison semantics:

- each row should distinguish:
  - expected vs unexpected
  - restored vs not restored

Current implementation rule:

- warning styling is applied when:
  - `expected = false`
  - or `restored = false`

Intended restore-control rule:

- `Re-restore` should only apply when a valid run and restore plan exist

## 14. Screen Boundary

The `dataDiff` screen is responsible for:

- one-run diff review
- expected/unexpected change visibility
- restore-result visibility
- future raw-diff and restore-retry actions

The `dataDiff` screen is not currently responsible for:

- run browsing
- execution launch
- full report rendering

## 15. Known Gaps and Review Items

Review items discovered while documenting:

- diff rows are backed by `GET /api/phase3/runs/{runId}/data-diff`, with fallback to synthetic local data when the API is unavailable.
- DB name is inferred from the first environment config item rather than a run-specific binding.
- `View raw JSON` is implemented via `GET /api/phase3/runs/{runId}/data-diff/raw`.
- `Re-restore` is implemented via `POST /api/phase3/runs/{runId}/restore/retry`.
- Raw diff data and restore-result data are deterministic mock when no real run artifacts exist.
- Re-restore does not trigger a real restore workflow — it records intent and returns ACCEPTED.
